# Seinfeld dataset-build — medium sprint
**Profile:** `thoughtful//medium` · **Robustness:** standard · **Vendor:** config default

Make the Seinfeld LoRA dataset pipeline genuinely good: human-gated, iteration-friendly, vocab-change-aware, and source-diverse. Build the generic `builtin.human_review` primitive in Astrid as part of the same sprint — it's the missing piece both for this project and for several future ones.

## Background

The Seinfeld scene generator demo is an LTX 2.3 LoRA + script + render project planned in `/project.md` (gitignored, repo root). One of four workstreams is `dataset_build` — a bucket-fill loop that collects training clips from YouTube and writes a captioned manifest. A v0 of that orchestrator runs end-to-end:

- Pack: `astrid/packs/seinfeld/` (gitignored — see `.gitignore`)
- Orchestrator: `astrid/packs/seinfeld/dataset_build/{orchestrator.yaml,run.py,STAGE.md}`
- Locked vocabulary: `astrid/packs/seinfeld/vocabulary.yaml`
- Hand-written schemas: `astrid/packs/seinfeld/schemas/{bucket_judge.json,caption.json}`
- v0 run output: `runs/seinfeld-dataset/` — 30 accepted clips (15 jerrys_apt + 15 monks_diner), provisional `manifest.json`, per-clip `.caption.json`s
- Existing tools extended this sprint: `builtin.youtube_audio` (now supports `--mode video` and `--url`) and `builtin.visual_understand` (now supports `--response-schema PATH`)

Read these before planning, in order:

## What v0 got right (do not change)

- Bucket-fill state file (`state.json`) — resumable forward
- Search query templates per bucket (`BUCKET_QUERIES`) inside `run.py`
- Per-video subdir with downloaded mp4 + scenes.json + per-scene `.judge.json`
- Cut clips + `.caption.json` land in `accepted/<bucket>/`
- VLM judge runs in `fast` mode, caption runs in `best` mode (cost discipline)
- `builtin.visual_understand --response-schema` is the right pattern for vocabulary-locked outputs — don't fork it

## What v0 got wrong (this sprint fixes)

Observed concretely on the v0 run:
1. **Source concentration.** 5 of 15 jerrys_apt clips came from one source video; multiple monks_diner clips from one compilation. The bucket-fill loop has no per-source cap.
2. **No human review step.** Caption errors and judge false-positives ship into the final manifest. Some captions describe a frame, not the motion in the clip. The reviewer is the natural place to catch this.
3. **Vocab drift is silent.** Editing `vocabulary.yaml` does nothing to existing artifacts — captions and judgements stay frozen against the old vocab. The user has no signal that a re-run is needed.
4. **Schemas are hand-written.** `bucket_judge.json` and `caption.json` were generated by Claude from `vocabulary.yaml`. The two will drift; the yaml is the only source of truth that should be edited.
5. **Rejection has no feedback loop.** Even if there were a reviewer, a "reject — watermark" decision doesn't inform the next collection round. The same source channel will be queried again next run.
6. **Phase-level rewind is manual.** To re-judge with a tighter prompt, the user has to delete every `*.judge.json` by hand AND clear `state.json` to un-process the video ids. There is no `--redo judge` or equivalent.

## What to build

Five primitives + one integration. All sized for ~1 week. Listed in dependency order, but the agent can re-sequence within reason.

### 1. `builtin.human_review` — generic human-gate executor

**Location:** `astrid/packs/builtin/human_review/{executor.yaml,run.py,STAGE.md}`

A single Python file (~120 LOC plus tests) that spins up a local HTTP server, opens the browser at a free port, and blocks until the page POSTs to `/submit`. Generic — must be reusable by any orchestrator that needs a human in the loop. Not Seinfeld-specific in any way.

Required CLI surface:
- `--html PATH` — single HTML file OR directory served as `/`
- `--data PATH` — JSON file, served as `/data.json` (read-only mount)
- `--serve PREFIX=DIR` — repeatable static mount (e.g. `--serve /clips=runs/seinfeld-dataset/accepted`)
- `--state PATH` — JSON file for partial saves; `POST /save` writes to it (per keypress, resumable)
- `--out PATH` — JSON file written on `POST /submit`; server exits 0 after write
- `--response-schema PATH` (optional) — JSON schema that validates `/submit` body strictly; mirror the `visual_understand --response-schema` contract
- `--port N` — explicit port, or `0` for auto (default 0)
- `--no-open` — skip auto-browser-launch (for headless CI / scripted use)
- `--timeout S` — exit nonzero if no submit by deadline; default unlimited

Contract details to decide during planning:
- How does `/save` interact with `/submit`? (Suggested: `/submit` body wins; `/save` is best-effort recovery state.)
- Should the server require a per-session token to reject drive-by POSTs from other localhost processes? (Suggested: yes — auto-generated, printed once on launch, embedded as a query param in the page URL.)
- How are errors surfaced to the page? (Suggested: HTTP 4xx with a JSON `{error, detail}` body the page can render.)

### 2. Seinfeld `review.html`

**Location:** `astrid/packs/seinfeld/dataset_build/review.html` (single file, inline JS/CSS, ~80–120 LOC)

Consumes `/data.json` (the provisional manifest), plays one clip at a time from `/clips/<bucket>/<id>.mp4`, displays caption + judge confidence below. Hotkey-driven, no clicks required for the happy path:
- `Y` / `Space` / `→` — accept, advance
- `N` / `X` — reject, advance
- `←` — go back
- `1`–`4` — tag a reject reason (`watermark | wrong_scene | wrong_character | bad_motion`); applies to most-recent reject
- `E` — open inline caption-edit overlay (saves a corrected caption alongside the decision)
- `Enter` — when all reviewed, `POST /submit` with the full decision set

Black background, large video, minimal chrome. Progress bar `X / N reviewed`. Decisions auto-`POST /save` on every keypress.

Output schema (validated by `--response-schema`):
```json
{
  "decisions": {
    "<clip_id>": {
      "decision": "accept" | "reject",
      "reject_reason": null | "watermark" | "wrong_scene" | "wrong_character" | "bad_motion",
      "edited_caption": null | "<string>",
      "reviewed_at": "<iso8601>"
    }
  },
  "completed_at": "<iso8601>"
}
```

### 3. Content-addressed artifact freshness

**Goal:** changing `vocabulary.yaml` or a judge prompt makes the pipeline re-run only the affected phases on the next invocation, without manual file deletion or a `--redo` flag.

**Approach:** each artifact (scenes.json, per-scene judge.json, per-scene caption.json, per-clip mp4) carries a small `.meta.json` sidecar recording the hash of its inputs. On re-run, before invoking a step, the orchestrator computes the expected hash from current inputs and compares — stale → regenerate.

Hash inputs per phase:
- `scenes.json`: hash(source video bytes) — almost never invalidates
- `judge.json`: hash(scene window + judge schema content + judge prompt template + vocab version)
- `caption.json`: hash(scene window + caption schema content + caption prompt template + vocab version)
- `cut clip mp4`: hash(source video bytes + scene start/end + ffmpeg encode options)

`vocab version` is a hash of `vocabulary.yaml` contents (skip whitespace/comments). The phase boundary is the level we hash at — don't go finer.

User-facing behavior on re-run after a vocab edit:
```
vocab: v1 (hash a8c3...) → v2 (hash f912...)
  judge.json stale: 30 → regenerate ($0.06 est)
  caption.json stale: 30 → regenerate ($0.45 est)
  scenes.json fresh, mp4 fresh — keep
proceed? [Y/n]
```

The cheap `--redo <phase>` flag is **NOT in scope** if content-addressing works — it becomes redundant. Drop it.

### 4. Vocabulary → schemas generator

**Location:** `astrid/packs/seinfeld/vocab_compile.py` (or wherever fits Astrid conventions — judgment call)

Reads `vocabulary.yaml`, writes `schemas/bucket_judge.json` and `schemas/caption.json`. Runs automatically as the first step of `dataset_build/run.py` so the schemas are always derived. `vocabulary.yaml` is the only file the user edits. Generated schemas carry a `// generated from vocabulary.yaml@<hash>` header.

Bucket-judge schema is derived from `scenes` + `characters`. Caption schema is derived from `scenes` + `characters` (with per-character outfit enums) + `shot_types` + the `caption_template` string.

### 5. Source-diversity cap + reject-reason feedback

**In `dataset_build/run.py`:**
- New flag `--max-clips-per-source N` (default 3). The bucket-fill loop tracks `clips_per_source_video[vid]` in `state.json` and stops accepting from a video once it hits cap, even if the bucket isn't full. This is a one-line change; the value is observable behavior.
- After human review completes, the orchestrator reads `human_review.final.json` and:
  - **Filters** the final manifest to only `decision == "accept"` clips
  - **Records** rejected source-video ids → `state.json["denylist_source_videos"]` (any reject with reason `watermark` or `wrong_scene` triggers this for the whole source video)
  - **Top-up:** if reject count brings any bucket under target, re-run the bucket-fill loop with denylist applied, then re-open the reviewer on the new clips only. Cap: 2 top-up rounds max per invocation.
- Reject reasons inform a single judge-prompt extension at the start of each iteration: lines like *"In prior rounds you frequently mis-accepted watermark overlays as valid Seinfeld scenes. Reject any frame with a channel logo or text overlay in the corners."* Build this from a small static map of reason → prompt-extension; not generated by an LLM.

### 6. Orchestrator integration

**In `dataset_build/run.py`:**
The driver should chain:
```
generate schemas → bucket-fill (with cap + denylist) →
  write provisional manifest → invoke builtin.human_review →
  read decisions → maybe top-up (≤2 rounds) →
  write final manifest
```

Plus modes:
- Default: run all phases as needed (content-addressing skips fresh ones automatically).
- `--review-only`: skip collection, open reviewer on existing provisional manifest. Useful for re-reviewing.
- `--no-review`: stop at provisional manifest. Useful for headless / CI.
- `--top-up-rounds N` — override the default 2 cap.

Resumability invariants (must hold across all modes):
- Interrupt mid-collection → re-run picks up at next video
- Interrupt mid-review → re-run opens reviewer at next unreviewed clip
- Interrupt during top-up → re-run resumes the top-up round
- Edit `vocabulary.yaml` mid-flight → re-run invalidates and re-runs affected phases only

## Contract decisions to make during planning

These are the judgment calls the premium planner should chew on rather than guess at:

1. **Where the human_review primitive's HTTP server lifecycle lives.** Foreground subprocess that blocks the orchestrator? Background daemon with a status command? (Suggested: foreground subprocess, blocks the calling orchestrator — matches Astrid's existing one-step-at-a-time mental model.)
2. **How content-addressed metadata is stored.** Sidecar `.meta.json` per artifact, or single per-video `manifest.meta.json`, or sqlite? (Suggested: sidecar; survives file moves, is human-readable, no schema migration.)
3. **Schema validation strictness on `/submit`.** Reject malformed submits with 4xx + retain pre-submit state, OR best-effort accept and log? (Suggested: strict 4xx with state retained, so the page can fix and retry.)
4. **How top-up rounds interact with the reviewer.** Re-open immediately after top-up collection, or batch top-ups and have one final review? (Suggested: re-open after each round so the human sees the effect of their last round of rejects — but cap at 2 rounds to avoid infinite loops.)
5. **Where the vocab-compile step runs.** Pre-step inside `run.py` every time, or a separate `seinfeld.vocab_compile` executor invoked from the orchestrator? (Suggested: pre-step; one less moving part. Promote to executor if a second orchestrator needs the same compile.)

## What's explicitly out of scope this sprint

Defer to a later sprint, even if tempting:

- Source adapters beyond YouTube (local folders, Bilibili, image directories)
- Cheap pre-VLM detectors (resolution, watermark, blackness, face-count)
- Live terminal dashboard / `astrid datasets status` command
- `astrid datasets new <name>` scaffolding command
- Multi-format manifest export (ai-toolkit, kohya, diffusers all-at-once)
- Parallelism (concurrent downloads, parallel VLM)
- Cost tracking / budget caps in real-time
- Cross-version trace + diff (`astrid datasets diff v1 v2`)
- Reviewer compare-mode (4-up grid) or bulk actions
- Migration of `dataset_build` onto Astrid task-mode (`task.yaml` plans + gates)

These are all real and worth doing. They're not load-bearing for the Seinfeld project specifically and would bloat this sprint past its 1-week budget.

## Acceptance criteria

The sprint is done when all of these are true:

1. **Reviewer works end-to-end on the existing v0 dataset.** Running `python3 -m astrid.packs.seinfeld.orchestrators.dataset_build.run --review-only --out runs/seinfeld-dataset` opens the browser, lets the user review all 30 clips with hotkeys, persists decisions to disk per keypress, exits cleanly on submit, writes `human_review.final.json`.
2. **`builtin.human_review` is genuinely generic.** A unit test or smoke test invokes it with a one-line stub `review.html` and `data.json` and verifies the round-trip works without any Seinfeld-specific code paths.
3. **`builtin.human_review` has `executor.yaml` + `STAGE.md` per Astrid conventions.** Visible in `python3 -m astrid executors list`. Inspectable via `python3 -m astrid executors inspect builtin.human_review --json`.
4. **Vocabulary edit triggers correct re-runs.** Add a third bucket to `vocabulary.yaml`. Re-run. Existing clips' judgements + captions invalidate and regenerate (with cost preview shown); scenes.json + mp4 files do not redownload or re-segment.
5. **Source diversity cap is enforced.** New bucket-fill run with `--max-clips-per-source 3` produces no more than 3 accepted clips from any one source video, even if the bucket would otherwise still need more.
6. **Reject-feedback closes the loop.** Reject 3 clips in review with reason=`watermark`. Re-run with top-up enabled. Those source videos are excluded; new candidates are queried; the bucket refills to target; second review opens on the new clips only.
7. **Final manifest contains only accepted-after-review clips.** No rejected clips appear in `manifest.json`. Each entry carries provenance (source_url, search query, judge confidence, reviewer decision, reviewer reason if any, edited caption if any).
8. **The seinfeld pack is still gitignored.** The `builtin.human_review` executor is **not** (it's a real Astrid primitive, belongs in the committed tree).

## Repo conventions to honor

- Generated files under `runs/` (gitignored)
- No secrets in code; use existing `load_api_key` from `generate_image/run.py` if you need OpenAI
- Astrid executors and orchestrators use `<pack>.<slug>` ids — `builtin.human_review` not bare `human_review`
- Tests under `tests/`; existing ones currently pass via `pytest --tb=no -q --no-header`
- Don't promote project-specific code (`review.html`, BUCKET_QUERIES, the reject-reason map) into `builtin/` — those stay under `seinfeld/`
- Local Python is via `PYENV_VERSION=3.11.11`; this is a peteromalley-only quirk and shouldn't be hardcoded into the executor
