The `tools/` directory is a small, file-based toolkit for turning a long source recording into a hype cut without adding a framework or pipeline runner. The current canonical flow is pool-based: source analysis runs once per source into `pool.json`, then each new `brief.txt` composes its own `arrangement.json`, cut artifacts, and optional render without rerunning the expensive LLM passes.

Each CLI still reads and writes plain files on disk so you can inspect intermediate JSON, run stages independently, and rerun only the step you need. The implementation lives in the `artagents/` package; root-level `*.py` files are compatibility launchers for existing `python3 pipeline.py ...` and `python3 cut.py ...` workflows. The top-level orchestrator is `pipeline.py --video SRC --brief BRIEF --out DIR`, which builds source-level artifacts in `DIR/` and brief-level artifacts in `DIR/briefs/<slug>/`.

## Pure-generative and no-audio modes

`pipeline.py` also supports an audio-backed no-video path:

```bash
python3 pipeline.py --audio rant.wav --brief brief.txt --out out --render
```

When `--video` is omitted, the pipeline skips source-derived visual analysis: `scenes`, `quality_zones`, `shots`, `triage`, `scene_describe`, `quote_scout`, and `pool_build`. It still transcribes `--audio`, always runs `pool_merge`, then runs `arrange`, `cut`, `render`, and validation when `--render` is set.

In this mode `cut.py --audio rant.wav` registers the rant audio under asset key `rant` and emits one full-length `clipType="media"` clip on `a1`. With no primary video asset, `hype.timeline.json` defaults to `1920x1080` at `30fps`. At least one effect folder must exist under `../effects/`; v1 ships `text-card`.

For visual-only generation, omit both `--video` and `--audio` and pass `--target-duration`:

```bash
python3 pipeline.py --brief brief.txt --theme 2rp --out runs/foo --render --target-duration 28
```

No-audio mode runs `pool_merge`, `arrange`, `cut`, and optionally `render`. It skips source-video analysis plus `transcribe`, `refine`, `editor_review`, and `validate`, and the emitted timeline has no `a1` track.

## Effects and themes

Shared render effects live at workspace root under `effects/<id>/`. Each effect folder must contain `component.tsx`, `schema.json`, `defaults.json`, and `meta.json`; `preview.mp4` is optional. The catalog is generated by scanning that directory, not by maintaining a hand-written list.

Styledoc themes live under `themes/<id>/`. `pipeline.py --theme arca-gidan` is the single entry point that threads the styledoc through `arrange`, `cut`, and `render`.

`theme.json` has five top-level blocks:

- `visual`: Remotion design tokens consumed by `useTheme()`, plus `canvas`, which drives `timeline.output` in pure-generative mode.
- `generation`: image/video generation prompts, model hints, aspect, `references[]`, and `assets[]`.
- `voice`: LLM tone, preferred/avoided lexicon, and overlay copy style for arrangement-authored text.
- `audio`: music and SFX intent for forward compatibility; no music-pick consumer is wired yet.
- `pacing`: fallback clip duration plus tempo hint for arrangement and generative materialization.

Each theme directory may contain:

- `references/`: style references for IP-Adapter or mood guidance.
- `assets/`: concrete reusable assets such as logos or recurring imagery.
- `effects/`: parent folder for project-only effects.
- `effects/<id>/`: a scoped effect folder with `component.tsx`, `schema.json`, `defaults.json`, and `meta.json`.

Workspace effects are scanned first, then active-theme effects. A theme effect overrides a workspace effect with the same id and prints a stderr warning, so effect ids stay stable while projects can customize behavior. The registry generator maintains `tools/remotion/_active_theme` as a symlink to the active theme directory, with `_active_theme.txt` as a Windows pointer-file fallback.

Canvas authority is one-way: `theme.visual.canvas` drives `timeline.output.resolution` and `timeline.output.fps` only in pure-generative mode, where there is no primary asset. Source-cut mode still derives output from the primary asset so `hype.timeline.json` and `hype.assets.json` remain byte-compatible with existing source-cut contracts.

After adding or renaming an effect, run:

```bash
cd remotion
npm run gen-types
```

This regenerates both `src/types.generated.ts` and `src/effects.generated.ts`.

## Reigh-app contract scope

Source-cut output preserves the existing shared contracts: legacy `clipType="text"` overlays still render, and source-cut timeline clips do not need the new generative `params` field. Pure-generative output intentionally uses extended v1 shapes such as `clipType="text-card"` and `params`; reigh-app does not consume those yet. Matching reigh-app schema sync and `<Player>` integration are accepted follow-up work.

## Duration enforcement

`timeline.validate_arrangement()` validates arrangement shape and pool references only. The 75-90 second source-cut duration window is now explicit opt-in through `timeline.validate_arrangement_duration_window()`.

Callers that need source-cut enforcement call the helper only when `timeline.is_all_generative_arrangement(arrangement, pool)` is false. All-generative arrangements may follow the rant audio duration instead, including short fixtures used by tests.

## Follow-ups

Out of scope for v1: render service or Railway worker, Supabase upload and asset rewrite, bndc Discord recap consumer, effects gallery on banodoco-website, npm package publishing for `effects/`, LLM-authored effect generation, reigh-app `<Player>` integration, and reigh-app schema sync for new clipType ids and `params`.

## `cut.py` outputs

`cut.py` now emits four artifacts into `--out`:

- `hype.edl.csv`: legacy CSV edit decision list for the existing overlap window.
- `hype.timeline.json`: reigh-app-compatible `TimelineConfig` output written via `timeline.save_timeline()`.
- `hype.assets.json`: reigh-app-compatible `AssetRegistry` output written via `timeline.save_registry()`.
- `hype.metadata.json`: tools-only provenance sidecar with pipeline metadata, clip provenance, and per-asset source metadata.

In arrangement mode, `hype.timeline.json` is multi-track by default:

- `a1`: dialogue and other audible spine clips.
- `v1`: the speaker-facing visual track aligned to the active dialogue span.
- `v2`: b-roll coverage and any `clipType="text"` overlay clips.

The committed schema anchors live in [`examples/`](examples/). `examples/hype.timeline.json`, `examples/hype.assets.json`, and `examples/hype.metadata.json` are round-trip checked through the Python load/save helpers.

## Pool-Based Flow

Canonical invocation:

```bash
python3 pipeline.py --video SRC --brief BRIEF --out DIR --render
```

The default pipeline order is:

1. `transcribe`
2. `scenes`
3. `shots`
4. `triage`
5. `scene_describe`
6. `quote_scout`
7. `pool_build`
8. `arrange`
9. `cut`
10. `refine`
11. `render`
12. `validate`

Artifact layout:

```text
DIR/
  transcript.json
  scenes.json
  shots.json
  scene_triage.json
  scene_descriptions.json
  quote_candidates.json
  pool.json
  logs/
  briefs/
    <brief_slug>/
      brief.txt
      arrangement.json
      hype.timeline.json
      hype.assets.json
      hype.metadata.json
      refine.json
      hype.mp4
      validation.json
      logs/
```

The pool is brief-agnostic. Re-running the same source with a different `--brief` should reuse `pool.json` and only execute `arrange`, `cut`, `refine`, `render`, and `validate` for the new brief slug.

## Brief Format

`brief.txt` is plain text. Keep it free-form, with one paragraph per beat or editorial requirement. The arranger hashes the copied `briefs/<slug>/brief.txt` contents into `brief_sha256`; if that hash changes for a given slug, only that brief directory is invalidated and rebuilt.

Typical brief guidance includes:

- the opening hook or dialogue spine you want emphasized
- the emotional arc or pacing target
- where b-roll should cover or contrast the speaker track
- any optional on-screen text overlays to include

## CLI Invariants

- `--brief` is required for the canonical `pipeline.py` flow.
- `--video` is required for source-derived steps; pure-generative mode may supply only `--audio`.
- `--arrangement` and `--pool` are the canonical `cut.py` inputs for the new flow.
- `--renderer ffmpeg-legacy` retains the Sprint 1 single-source ffmpeg path for one release and is deprecated.
- `--asset KEY=PATH` is repeatable and defines additional source assets for the arrangement-driven flow.
- `--verbose` enables ffprobe cache tracing and prints `ffprobe RUN <key>` or `ffprobe SKIP <key>`.

## Multi-track Rendering

The rendered timeline stays byte-compatible with the reigh-app schema and uses existing track types:

- `a1` carries the dialogue spine or other audible clips selected by the arrangement.
- `v1` carries the speaker-facing visual track for that moment.
- `v2` carries b-roll coverage when it differs from `v1`, plus any optional text overlays.

Text overlays are emitted as ordinary `clipType="text"` clips on `v2`. The validator-facing metadata contract is unchanged: overlay clips still route through `caption_kind="visual"` with no `source_transcript_text`.

## Migration from `--picks`

`cut.py --picks ...` still works for the legacy flow, but it emits a `DeprecationWarning`. The upgrade path is:

1. build a source-level `pool.json`
2. compose a brief-level `arrangement.json`
3. call `cut.py --pool pool.json --arrangement arrangement.json --brief brief.txt`

## Resume mode

Use `--timeline` to resume from an existing `hype.timeline.json` without re-running pick parsing, ranking, or scene loading:

```bash
python3 cut.py --timeline out/hype.timeline.json --out out2 --render
```

Resume mode guarantees:

- `hype.timeline.json` is byte-round-tripped through `timeline.load_timeline()` / `timeline.save_timeline()`.
- `hype.metadata.json` is re-emitted and carries forward prior clip entries for clip ids that still exist in the timeline.
- `hype.assets.json` stays byte-identical when `--out` matches the timeline source directory.
- Relative asset paths are rebased to absolute source paths when `--out` differs from the timeline source directory.
- Only `--renderer remotion` is supported when `--render` is requested.

These flags are rejected with `--timeline`: `--scenes`, `--video`, `--picks`, `--shots`, `--transcript`, `--top-k`, `--primary-asset`, `--asset`, `--output-resolution`, and `--output-fps`.

## Audio model

`remix.py` has been removed. The audio model now lives entirely in `hype.timeline.json`:

- Per-clip audio uses `clip.volume`.
- Per-track audio uses `track.muted` and `track.volume`.
- Remotion rendering honors this natively via `@remotion/media`.

Background music is just another audio-track clip in the timeline JSON. For example, `-18 dB` is approximately `volume: 0.126`. There is no ducking, sidechain compression, or other automatic audio level management in Sprint 3.

## Remotion Project

Sprint 2 adds a standalone Remotion project in [`remotion/`](remotion/). It is intentionally isolated from the reigh-app workspace and bundles only the clean-room files committed under `tools/remotion/`.

Layout:

- [`remotion/package.json`](remotion/package.json): pinned Remotion, React, and npm scripts.
- [`remotion/src/types.generated.ts`](remotion/src/types.generated.ts): generated from [`timeline.py`](timeline.py) via `npm run gen-types`.
- [`remotion/src/lib/`](remotion/src/lib): clean-room ports of the trim, duration, track, fade, and viewport layout helpers.
- `remotion/src/*.tsx`: the upcoming composition surface that consumes `TimelineConfig` plus `AssetRegistry`.

Before using the Remotion renderer, install Node dependencies once:

```bash
cd remotion
npm install
```

Useful targets:

- `npm run gen-types`
- `npm run typecheck`
- `npm run smoke`

If you extend any of the allowed-key frozensets in [`timeline.py`](timeline.py), re-run `npm run gen-types` and update `examples/hype.timeline.full.json` in the same PR. The schema-contract test will fail otherwise.

`anthropic>=0.40.0`, `google-genai>=1.15.0`, and `python-dotenv>=1.0.0` are required for the pool-building LLM stages and their mocked contract tests.

## `--primary-asset`

`--primary-asset` tells `cut.py` which asset the `--scenes`, `--transcript`, and `--shots` inputs describe.

- If `main` exists in the resolved asset map, `--primary-asset` defaults to `main`.
- If the active arrangement uses multi-source assets without a `main` asset key, `--primary-asset` is required.
- The primary asset drives the default `output.resolution` and `output.fps` values in `hype.timeline.json`.
- The primary asset also receives the derived `scenes_ref`, `transcript_ref`, and `shots_ref` fields in `hype.metadata.json`.

## `metadata.sources`

`hype.metadata.json` is the only tools-specific file. `hype.timeline.json` and `hype.assets.json` stay byte-compatible with the reigh-app schemas and do not carry extra fields.

Within `metadata.sources`:

- `codec` is ffprobe-derived per asset.
- `scenes_ref`, `transcript_ref`, and `shots_ref` attach only to `sources[primary_asset]`.
- `codec` survives reruns on cache hits; `*_ref` fields do not — they are always re-derived from CLI args each run.

## Rendering

Use `cut.py --render --renderer remotion` for the Sprint 2 default path. It writes `hype.timeline.json`, `hype.assets.json`, `hype.metadata.json`, and then asks [`render_remotion.py`](render_remotion.py) to produce `hype.mp4` from the standalone Remotion project.

The renderer flag surface is `cut.py --render --renderer {remotion, ffmpeg-legacy}`.

```bash
python3 cut.py \
  --scenes scenes.json \
  --transcript transcript.json \
  --pool pool.json \
  --arrangement arrangement.json \
  --brief brief.txt \
  --asset main=main.mp4 \
  --asset broll=broll.mp4 \
  --primary-asset main \
  --render \
  --renderer remotion \
  --out out
```

`--renderer ffmpeg-legacy` is still available as a deprecated fallback for one release. That path preserves the Sprint 1 subprocess ffmpeg behavior and still assumes a single render source asset.

## `pipeline.py`

`pipeline.py` is the top-to-bottom orchestration wrapper for the pool-based tools chain:

- `transcribe -> scenes -> shots -> triage -> scene_describe -> quote_scout -> pool_build -> pool_merge -> arrange -> cut -> refine -> render -> validate`

CLI surface:

- `--video`, `--brief`, and `--out` are the canonical source-cut inputs.
- `--audio` defaults to `--video`.
- `--audio`, `--brief`, and `--out` are the canonical pure-generative inputs.
- `--brief`, `--out`, and `--target-duration` are the canonical no-audio visual-only inputs.
- `--theme` forwards a theme JSON path to `render_remotion.py`.
- `--plan` is deprecated compatibility input. It maps to `--brief` and emits a `DeprecationWarning`.
- `--brief-slug` defaults to `Path(--brief).stem`.
- `--source-slug` defaults to `Path(--out).name`.
- `--asset KEY=PATH` is repeatable and is forwarded to `cut.py`.
- `--primary-asset` is forwarded to `cut.py`.
- `--from STEP` forces that step and everything after it to rerun.
- `--skip STEP` removes a step from the chain entirely.
- `--render` enables `render_remotion.py` and the follow-on `validate.py` step for audio-bearing modes. Without it, `pipeline.py` stops after `cut.py` and logs that validate was skipped.
- `--target-duration` sets the output duration for no-audio runs; audio-backed pure-generative runs still derive arrangement duration from `--audio`.
- `--config` accepts JSON, or YAML only when `PyYAML` is installed.
- `--python` overrides the interpreter used for child scripts.
- `--verbose` streams each child process log to the parent terminal while still writing log files.

Step sentinels:

- `transcribe`: `transcript.json`
- `scenes`: `scenes.json`
- `shots`: `shots.json`
- `triage`: `scene_triage.json`
- `scene_describe`: `scene_descriptions.json`
- `quote_scout`: `quote_candidates.json`
- `pool_build`: `pool.json`
- `pool_merge`: always runs and upserts generative `pool_g_*` entries into `pool.json`
- `arrange`: `briefs/<slug>/arrangement.json`
- `cut`: `briefs/<slug>/hype.timeline.json`, `briefs/<slug>/hype.assets.json`, `briefs/<slug>/hype.metadata.json`
- `refine`: `briefs/<slug>/refine.json`
- `render`: `briefs/<slug>/hype.mp4`
- `validate`: `briefs/<slug>/validation.json`

Caching rules:

- A step is a cache hit only when every sentinel for that step already exists.
- Per-source sentinels live directly under `--out`.
- Per-brief sentinels live under `--out/briefs/<brief_slug>/`.
- Changing `brief.txt` for one slug invalidates only that brief directory, never the shared per-source artifacts.
- `cut` is treated as a single three-file unit. If only part of a prior `cut.py` run exists, `pipeline.py` prints `cut: partial prior output detected, rerunning` and reruns `cut.py`.
- `--from STEP` ignores cache state for that step and every later step.

Operational notes:

- Per-source logs live at `<out>/logs/<step>.log`.
- Per-brief logs live at `<out>/briefs/<slug>/logs/<step>.log`.
- `--skip shots` is the escape hatch for avoiding the still-frame extraction pass. Today, `cut.py` only uses `--shots` to populate the metadata-sidecar provenance reference (`metadata.sources[primary_asset].shots_ref`); it does not change clip selection or render output.
- `render` is intentionally separate from `cut`. `pipeline.py` calls [`render_remotion.py`](render_remotion.py) directly so `hype.mp4` can be cached independently of the three `cut` artifacts.
- `--config` may include an `extra_args` object shaped like `{"extra_args": {"scenes": ["--threshold", "24"], "render": ["--composition", "HypeComposition"]}}`.

## `open_in_reigh.py`

`open_in_reigh.py` is a best-effort manual handoff helper. It does not talk to Supabase directly and it does not modify the reigh-app repository beyond optional local file copies you explicitly ask it to write.

Storage model and scope:

- reigh-app is Supabase-backed. The live editor reads `public.timelines` rows with `config` and `asset_registry` columns.
- The current provider lives at `reigh-app/src/tools/video-editor/data/SupabaseDataProvider.ts`.
- Because the editor is row-backed rather than file-backed, `open_in_reigh.py` is primarily a handoff tool, not an importer.

Handoff modes:

- `--copy-to DIR` writes byte-preserved copies of `hype.timeline.json` and `hype.assets.json` wherever you want them.
- Without `--copy-to`, the helper opportunistically probes `public/timelines/`, `public/demos/`, `timelines/`, and `demos/` under `--reigh-app`, then copies into the first existing location it finds.
- If none of those directories exist, it prints a manual-handoff block describing the Supabase storage model and the next steps instead of failing.
- `--print-sql` prints an `INSERT ... ON CONFLICT (id) DO UPDATE` template for `public.timelines`.
- `--timeline-id` is required. There is no default. You must supply a real UUID yourself.
- The SQL template intentionally leaves `<PROJECT_ID>` and `<USER_ID>` placeholders for you to fill. The bridge never opens a Supabase connection or writes rows on your behalf.

Asset limitation:

- `hype.assets.json` contains local absolute paths from `cut.py`.
- reigh-app resolves asset `file` values as HTTP URLs or `timeline-assets` bucket keys, not local filesystem paths.
- Before a handed-off timeline will play inside reigh-app, you must upload the media to the `timeline-assets` bucket or self-host it, then update each asset `file` value to the resulting bucket key or HTTP URL.

## Top-to-bottom example

```bash
python3 pipeline.py \
  --video examples/main.mp4 \
  --brief brief.txt \
  --out out \
  --render

python3 open_in_reigh.py \
  --out out \
  --copy-to /tmp/reigh-handoff \
  --timeline-id 123e4567-e89b-12d3-a456-426614174000

python3 open_in_reigh.py \
  --out out \
  --print-sql \
  --timeline-id 123e4567-e89b-12d3-a456-426614174000
```
